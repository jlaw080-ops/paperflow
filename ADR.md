# Architecture Decision Records

## 철학
- **MVP 속도 최우선.** 화려한 기능보다 "가져오기 → 편집 → 공유 → A4 인쇄"가 끝까지 도는 것이 먼저다.
- **외부 의존성 최소화.** 이미 연결된 도구(Supabase, Vercel)를 쓰고, 새 라이브러리는 꼭 필요할 때만.
- **표준 우선.** Obsidian 전용 문법이 아니라 표준 마크다운을 따른다.

---

### ADR-001: 문서 저장소로 Supabase를 쓴다
**결정**: 문서를 Supabase 데이터베이스에 저장한다.
**이유**: "앱 안에서 편집"하려면 저장 공간이 필요하다. Supabase는 이미 연결돼 있고, 로그인(Auth)과 보안(RLS)까지 한 번에 제공한다.
**트레이드오프**: 문서가 Github 파일이 아니라 DB에 살게 되므로 Obsidian과의 자동 동기화는 포기한다(→ ADR-002). 대신 단순함과 편집 편의를 얻는다.

### ADR-002: Obsidian과는 1회성 가져오기(import)만 한다
**결정**: Obsidian 내용은 `.md` 업로드로 한 번 가져오고, 이후 원본은 앱이 된다. 양방향 동기화는 하지 않는다.
**이유**: 로컬 파일과 웹 DB의 실시간 양방향 동기화는 충돌 처리가 매우 복잡하다(같은 문서가 양쪽에서 수정되면 어느 게 진짜인지 꼬인다). MVP에 맞지 않는다.
**트레이드오프**: Obsidian에서 나중에 고친 내용이 앱에 자동 반영되지 않는다. 대신 구조가 단순하고 버그가 적다.

### ADR-003: 마크다운 렌더링은 react-markdown + remark-gfm
**결정**: 마크다운→HTML 변환에 `react-markdown`과 `remark-gfm`을 쓴다.
**이유**: 표·체크박스 등 표준 마크다운(GitHub Flavored Markdown)을 안정적으로 지원한다. 직접 파서를 만들 이유가 없다.
**트레이드오프**: Obsidian 전용 문법(`[[위키링크]]` 등)은 기본 지원되지 않는다. 이는 PRD의 MVP 제외 사항과 일치하므로 문제없다.

### ADR-004: A4 인쇄는 CSS(@page/print)로 구현한다
**결정**: 별도 PDF 생성 서버나 라이브러리 없이, `@media print` + `@page { size: A4 }` 인쇄 CSS로 페이지를 나눈다. 사용자는 브라우저 인쇄에서 "PDF로 저장"을 쓴다.
**이유**: 가장 단순하고 의존성이 없으며, 요구사항("화면은 이어지고, 인쇄/PDF는 A4 분할")을 그대로 충족한다.
**트레이드오프**: 픽셀 단위로 정밀한 페이지 레이아웃 제어는 제한적이다. 하지만 기술 문서 수준에서는 충분하고, 필요해지면 나중에 서버 렌더링 방식으로 교체할 수 있다.

### ADR-005: 열람은 완전 공개, 편집은 단일 소유자
**결정**: `/view/[slug]` 공개 라우트는 로그인 없이 읽기 전용. 편집·트리 관리는 소유자 로그인(Supabase Auth) 후에만. DB 차원의 RLS로 강제한다.
**이유**: 요구사항이 "누구나 링크로 열람 + 본인만 편집"이다. 권한을 앱 코드가 아니라 DB(RLS)에서 막아야 우회가 안 된다.
**트레이드오프**: 문서별 개별 권한(이건 공개, 저건 비공개)은 MVP에서 없다. 전부 공개로 통일한다.

### ADR-006: 폴더와 문서를 한 테이블로 표현한다
**결정**: `documents` 테이블에 `type('folder'|'doc')`과 `parent_id`를 두어 트리를 만든다.
**이유**: 폴더와 문서를 따로 두는 것보다 이동·정렬·조회가 단순하다.
**트레이드오프**: 한 테이블에 두 종류가 섞여 쿼리 시 `type` 구분이 필요하다. 트리 깊이가 매우 깊어지면 조회 최적화가 필요할 수 있으나, 개인 문서 규모에서는 문제없다.
